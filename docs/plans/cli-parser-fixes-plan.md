## Design decision needed (not covered by this plan)

The reporter's 2026-04-19 06:06 follow-up on #160 proposes two adjacent
enhancements at `1f8c5fab` for content with **literal unescaped internal
quotes** (e.g., LLM-written markdown citations):

- **Heredoc blocks** — `<<<...>>>` and `<<NAME...NAME` preprocessed before
  tokenization. Raw contents flow opaquely through the parser.
- **Greedy fallback** — when a segment has ≥2N non-escaped `"` and no
  unquoted flags, re-scan from quote pairs and let the last positional
  consume from the Nth `"` to the last `"`.

This is new user-facing syntax and a heuristic recovery path. It's materially
larger in scope than the other fixes and the reporter has flagged it as "open
questions for design feedback". **Recommend deferring** — merge the
characterization fixes first, evaluate the heredoc/greedy proposal against
actual LLM-produced failure rate afterward. If we want it, the greedy fallback
is the lower-risk of the two (single narrow code path, no new syntax to teach).

## Gap analysis — what the reference impls don't cover

Spot-check against the current codebase turned up one case the reporter's
fixes don't address:

- **`parseCliForDisplay`** (`ToolCliNormalizer.ts:223-246`) is a separate
  classifier used for streaming-phase UI preview. The reporter's #160 fix
  updates it to use `tokenizeWithMeta` + `wasQuoted`, but I should verify
  manually during port that the other fixes (A.4 unescape, D.2 empty token,
  Bug #2 array JSON) also reach this display path correctly. The audit tests
  exercise `parseCommandSegment` only — display-path regressions would not
  show up there.

Action: add 2-3 display-path cases to `ToolManagerCliSyntax.test.ts` during
PR 2 to cover this.

## Open questions for the maintainer

1. PR 1 is a 1-line logic change + 2-line defaulting helper. Should the fix
   also extend beyond `workspaceId` to `sessionId` / `memory` / `goal`? The
   issue only reproduces for `workspaceId`, but the same envelope-vs-params
   asymmetry exists for the others. My read: no — those are currently passed
   through the session + memory service layers, not via `params`. But worth
   confirming before PR 1 lands so we don't have to revisit.
2. PR 2 cherry-picks the reporter's commits verbatim. Do we want to
   re-author them (squash + our own commit message / sign-off), or preserve
   the reporter's authorship? Either works; question is only about giving
   credit. (My default: preserve authorship + add `Co-authored-by` if
   squashed.)
3. Heredoc (`<<<...>>>`) vs alternatives the reporter considered (fenced
   ` ``` `, `--content @file`, base64 flags). If we do pursue a solution for
   literal-internal-quote content, which syntax? This can wait for PR 3.

## Non-goals

- Do not rewrite the parser into a general shell language. The audit stays
  inside the existing `tokenize → parseCommandSegment` structure.
- Do not remove per-tool JSON schemas as source of truth for validation.
- Do not refactor `ToolBatchExecutionService.applyContextDefaults` beyond the
  defaulting extension in PR 1 (the two tool-specific branches stay as-is —
  they project additional context fields and should remain explicit).
